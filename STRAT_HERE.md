# 24 小时 AI 项目挑战赛工作流

本文件是比赛当天的执行手册。根目录 `AGENTS.md` 负责约束工程规范和交接规则，本文件负责把 24 小时里的决策、开发、部署、提交和 Demo 视频串成可执行流程。

## 0. 已知比赛要求

- 题目公布时间：2026-05-24 08:00。
- 最终提交通常至少包括：
  - public GitHub 仓库；
  - 可公网访问的产品 URL；
  - 不超过 3 分钟的 Demo 视频；
  - 可被评审登录查看运行环境的云服务器。
- PDF 准备说明强调：代码提交记录和最终部署时间会被查看，截止后构建和部署会被认定为超时完成。

## 1. 总体策略

- 先做能跑通的完整闭环，再做亮点。
- 所有产品想法都要落到一个浏览器可访问的交互产品上。
- 默认选择熟悉、可部署、文档充分的技术栈。
- 不为了炫技引入高风险依赖，不把核心功能押在不稳定服务上。
- 每 2 到 3 小时留一次可演示版本，避免最后只有半成品。

## 2. 推荐技术栈

题目公布前不提前绑定具体产品，但建议默认使用下面的低风险栈：

- Web 应用：Next.js 或 Vite + React。
- 后端接口：Next.js API route、Express，或同仓轻量 Node 服务。
- 样式：Tailwind CSS 或普通 CSS，优先少配置。
- LLM：服务端读取 API Key，前端只调用本项目 `/api/*`。
- 部署：云服务器 Node.js + npm + PM2。
- 数据：优先内存、JSON、SQLite 或浏览器本地存储；除非题目强需求，不引入复杂数据库。

如果题目明显需要 Python、多模态或模型推理，再按实际需求创建子项目和 guide。

## 3. 24 小时节奏

### 0:00 - 0:45 读题和定范围

- 读题后写一句话产品定义：谁在什么场景下，用它完成什么任务。
- 定 MVP：3 个以内核心功能，必须能演示。
- 写清楚不做什么，避免范围膨胀。
- 创建或更新 README 的项目简介和运行方式草稿。

### 0:45 - 3:00 搭项目骨架

- 初始化项目、依赖、脚本和 `.env.example`。
- 跑通 `npm install`、`npm run dev`。
- 做第一版首页或主工作台，不做落地页空壳。
- 如果需要 LLM，先打通最小 API 调用和错误提示。

### 3:00 - 8:00 做可用 MVP

- 完成主流程：输入、处理、输出、重试或错误状态。
- 前端使用相对 API 路径，例如 `/api/chat`。
- 后端缺少环境变量时返回清晰错误。
- 开始积累 README 的真实命令，不等最后补。

### 8:00 - 14:00 做差异化亮点

- 补一个能打动评审的体验点：可视化、对话记忆、多模态、报告生成、工作流编排、行业化模板等。
- 每新增依赖都同步 `package.json` 和锁文件。
- 如果引入文件上传、缓存、截图、录制，确认临时产物进入 `.gitignore`。

### 14:00 - 18:00 部署和生产构建

- 本地先跑 `npm run build`。
- 服务器按 README 复现：
  - `git clone <repo-url>`
  - `npm install`
  - `cp .env.example .env.local`
  - 填写真实密钥
  - `npm run build`
  - `npm start` 或 PM2 启动
- 记录公网 URL、端口、安全组、防火墙状态。
- 不在仓库中提交真实 `.env.local`。

### 18:00 - 21:00 Demo 视频

- 准备 3 分钟以内脚本：
  - 15 秒：问题和目标用户；
  - 30 秒：核心设计思路；
  - 90 秒：完整产品演示；
  - 30 秒：技术实现和部署状态；
  - 15 秒：总结亮点。
- 用真实公网 URL 演示一次，避免只演示本地。

### 21:00 - 24:00 收口和缓冲

- 跑最终 `git status --short`，确认 `.env.local` 未被跟踪。
- README 补齐：技术栈、本地运行、环境变量、构建、生产启动、PM2、公网端口、FAQ。
- 新增最终 handoff，客观记录已验证和未验证内容。
- 留出提交、上传视频、填写表单和服务器复查时间。

## 4. 每次让 AI 工具接手时的固定动作

开头固定要求：

```text
请先读取 AGENTS.md、docs/handoffs/README.md、docs/handoffs/ 下最新交接记录，并运行 git status --short。不要执行 git add/commit/push。改完后新增 handoff，记录实际验证结果。
```

如果涉及子项目，再要求读取该子项目的 `Agents_for_this.md` 和相关 guide。

## 5. 代码和部署检查清单

- `package.json` 有 `dev`、`build`、`start`。
- `.env.example` 覆盖所有必需环境变量，且没有真实值。
- 前端不出现真实 API Key，也不把 API Key 传给浏览器。
- 生产接口不硬编码 `localhost`。
- `PORT` 可通过环境变量覆盖。
- README 命令能在干净服务器复现。
- `npm run build` 在本地通过；部署前在服务器也通过。
- PM2 命令和项目名写在 README。
- Demo 视频使用公网 URL 演示。

## 6. 建议目录结构

实际题目确定后再创建工程目录。若没有特殊要求，推荐：

```text
.
├─ AGENTS.md
├─ STRAT_HERE.md
├─ README.md
├─ .env.example
├─ .gitignore
├─ docs/
│  └─ handoffs/
├─ challenge/
└─ <app-project>/
```

如果实际应用放在子目录，README 必须明确 `cd <app-project>` 后再执行安装、构建和启动命令。

## 7. 决策原则

- 可演示优先于完美架构。
- 可部署优先于本地炫技。
- 清晰错误优先于静默失败。
- 文档同步优先于最后补救。
- 用户价值优先于模型堆料。
