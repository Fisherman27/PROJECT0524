# Interview Replay 升级方案

> 项目中文名：研答 Replay / 保研面试复盘教练  
> 项目英文名：Interview Replay  
> 文档定位：在现有“面试前模拟 + 面试后复盘”双模式基础上，进一步提升产品深度，避免项目停留在“几个输入框 + 一个大 Prompt”的层面。

---

## 1. 升级背景

当前项目已经形成一个比较清晰的基础定位：

> 面向保研学生的文字版 AI 面试复盘工具，通过“面试前模拟”和“面试后复盘”两种模式，帮助用户诊断回答中的表达损失、证据缺失、逻辑漏洞和导师追问风险，并生成下一次可复用的回答策略。

基础版本的核心价值是：

1. 面试前模拟：通过“临场回答 vs 冷静回答”的对比，帮助学生发现紧张状态下丢失了哪些关键信息；
2. 面试后复盘：通过“真实回答 vs 事后回答 vs 建议回答”的对比，帮助学生判断哪种表达更稳、更适合保研面试；
3. 输出结构化报告：包括问题意图、回答诊断、追问风险、最佳融合回答和可迁移回答公式。

但是，如果产品只是简单地把用户输入交给一个大模型，并让大模型生成一份点评报告，那么它仍然可能被认为是：

```text
Prompt 工程包装
ChatGPT 套壳
普通回答润色器
AI 模拟面试工具的轻量变体
```

为了让项目更有产品价值、技术表达和差异化，需要进一步升级。

升级的核心目标不是“堆功能”，而是把项目从：

```text
用户输入回答 → AI 直接点评
```

升级为：

```text
用户输入材料与回答
↓
系统结构化理解用户材料
↓
系统从多个视角诊断回答
↓
系统识别回答差距、材料遗漏和风险点
↓
系统生成下一次可复用的训练策略
```

也就是说，升级后的产品应该更像一个：

> **结构化保研面试复盘引擎**

而不是一个普通聊天式 AI 工具。

---

## 2. 总体升级方向

本项目可以从两条主线升级：

1. **轻量多角色诊断链**  
   借鉴多 Agent 思路，但不追求复杂框架，而是将复盘过程拆成多个明确角色：问题意图分析、材料证据匹配、导师风险审查、临场差距诊断、回答融合重构和训练建议生成。

2. **结构化复盘系统**  
   这是多 Agent 的输出层和产品层：从数据结构、诊断指标、用户材料利用、复盘可视化和训练闭环等方面增强产品，让产品不只是生成点评，而是形成可解释、可积累、可训练的复盘机制。

这两条路线并不冲突。实际项目中可以采用：

```text
底层：结构化复盘系统
上层：轻量多角色诊断链
```

也就是：先建立材料证据库、评分矩阵、风险雷达、复盘卡片等结构化机制，再让不同“角色”围绕这些结构输出诊断结果。

---

# 第一部分：轻量多角色诊断链

## 3. 为什么考虑多 Agent / 多角色思路

保研面试复盘不是单一维度任务。

同一道回答，至少需要从以下视角判断：

1. 这道题真正想考什么？
2. 用户有没有正面回答？
3. 用户有没有使用自己的简历和项目证据？
4. 用户有没有体现科研潜力？
5. 用户有没有过度包装？
6. 这段回答会不会引发危险追问？
7. 这段回答如何改得更稳、更自然？
8. 下一次应该练什么？

如果只用一个大 Prompt 一次性完成所有事情，容易出现几个问题：

- 分析维度混杂；
- 输出看似完整但不够具体；
- 容易直接生成“漂亮答案”，忽略真实性风险；
- 缺少不同视角之间的互相制衡；
- 很难解释产品为什么比 ChatGPT 更有价值。

因此，可以把复盘流程拆成多个角色。这里说的“多 Agent”不一定要使用复杂外部 Agent 框架；在本项目 P0 中，推荐使用项目内轻量 Agent Runner，通过多次 LLM 调用和独立结构化输出来实现真实多角色诊断链。

核心思想是：

> **让不同角色分别从导师视角、材料视角、表达视角和训练视角审查同一个回答，最终形成一份结构化复盘报告。**

---

## 4. 多角色诊断链总流程

升级后的诊断链可以设计为：

```text
用户输入：背景材料 + 面试问题 + 回答版本
↓
材料分析器：提取用户材料中的可用证据
↓
问题意图分析器：判断这道题真正考察什么
↓
材料证据匹配器：判断回答是否调用了合适材料
↓
导师风险审查员：识别空泛、夸大、追问风险
↓
临场差距诊断器：比较临场回答和冷静回答的差距
↓
回答重构器：生成最佳融合回答
↓
训练规划器：输出救场模板、回答公式和下一题建议
↓
最终报告生成器：整合所有结果，形成可读复盘报告
```

这条链路的价值在于：

- 产品不再只是“AI 生成点评”；
- 每一步都有明确任务；
- 用户能理解每个诊断结论从哪里来；
- 复盘结果更具体、更可执行；
- Demo 时更容易讲清楚“我们做了哪些关键取舍”。

---

## 5. 角色一：Question Intent Analyst 问题意图分析器

### 5.1 角色定位

问题意图分析器负责判断一道面试题背后的真实考察目标。

很多学生答不好，不是因为语言差，而是因为一开始就没有理解题目真正想问什么。

例如问题：

```text
你为什么选择我们实验室？
```

学生可能以为是在问：

```text
你是否感兴趣？
```

但保研导师真正可能在看：

```text
你是否了解实验室方向？
你的经历是否和这里匹配？
你是不是海投模板？
你有没有具体研究动机？
你是否具备长期科研潜力？
```

### 5.2 输入

```text
面试问题
面试类型
目标方向
目标院校/实验室，可选
```

### 5.3 输出

```text
这道题的真实意图
主要考察能力
常见答偏方式
理想回答应包含的信息层次
```

### 5.4 示例输出

问题：你为什么选择这个研究方向？

输出：

```text
这道题不是单纯考察“兴趣”，而是在判断：
1. 用户的兴趣是否来自真实经历；
2. 用户是否能说出具体研究问题；
3. 用户的本科经历是否和目标方向形成连续性；
4. 用户是否具备长期投入该方向的动机；
5. 用户是否只是泛泛地说“AI 很热门”。
```

### 5.5 产品价值

这个角色帮助用户从“我要怎么说得好听”转向“我到底应该回答什么”。

---

## 6. 角色二：Evidence Mapper 材料证据匹配器

### 6.1 角色定位

材料证据匹配器负责从用户材料中找出适合支撑当前问题的证据。

保研学生往往不是没有经历，而是不知道某道题应该使用哪段经历。

### 6.2 输入

```text
用户简历材料
科研/项目经历
自我介绍
个人陈述
面试问题
目标方向
```

### 6.3 输出

```text
这道题最适合调用哪些个人材料
哪些材料可以作为核心证据
哪些材料不适合在这道题使用
哪些材料如果使用需要补充准备
```

### 6.4 示例

用户材料：

```text
我做过一个基于大模型的论文中译英 TeX 编辑器，负责 Prompt 设计、前后端开发和翻译质量优化。
```

问题：

```text
你为什么想继续做大模型与科研写作相关方向？
```

证据匹配器输出：

```text
适合调用的材料：
1. 大模型论文翻译项目：可作为兴趣来源；
2. Prompt 设计经历：可证明不是泛泛接触；
3. 翻译质量优化：可引出科研写作场景中的可靠性问题。

需要谨慎的材料：
1. 如果提到“模型优化”，需要说明具体优化对象；
2. 如果提到“效果提升”，需要准备评价指标；
3. 如果提到“科研写作可靠性”，需要能解释具体问题。
```

### 6.5 产品价值

这个角色使产品拥有“材料感”。它不是孤立点评回答，而是判断用户有没有用好自己的材料。

## 7. 角色三：Skeptical Professor 导师风险审查员

### 7.1 角色定位

导师风险审查员负责站在严格导师视角挑刺，识别回答中的风险点。

它不负责温和鼓励，也不负责生成漂亮答案，而是专门判断：

```text
这句话会不会被追问？
这个表述是不是太空？
这个贡献是不是说大了？
这个术语用户能不能承接？
这里有没有证据不足？
```

### 7.2 输入

```text
面试问题
用户回答
用户材料
目标方向
```

### 7.3 输出

```text
高风险表述
可能追问
真实性风险
证据不足点
过度包装点
建议降风险表达
```

### 7.4 示例

用户回答：

```text
我主要负责模型优化，最后效果提升明显。
```

风险审查员输出：

```text
高风险点：
1. “模型优化”过于笼统，导师可能追问优化了哪个模型、哪个模块；
2. “效果提升明显”缺少指标，导师可能追问 baseline 和评价方式；
3. 没有说明个人贡献边界，容易被质疑是否是团队成果；
4. 如果无法说明实验设计，这句话会显得过度包装。

建议改法：
将“模型优化”改为更具体的工作，例如“我主要负责 Prompt 模板设计和翻译结果评估”；如果没有明确指标，不要说“显著提升”，可以说“我们主要从术语一致性和格式保持两个维度进行人工评估”。
```

### 7.5 产品价值

这个角色能强化产品的保研场景特征。保研面试最怕的不是回答不漂亮，而是说了自己无法承接的话。

## 8. 角色四：Performance Gap Diagnoser 临场差距诊断器

### 8.1 角色定位

临场差距诊断器专门用于“面试前模拟模式”。

它比较：

```text
临场回答
冷静回答
用户背景材料
```

然后判断用户紧张状态下丢失了哪些信息。

### 8.2 输入

```text
面试问题
临场回答
冷静回答
用户材料
```

### 8.3 输出

```text
临场损失类型
冷静回答补充的信息
仍然缺失的信息
下次避免崩掉的表达结构
```

### 8.4 临场损失分类

| 损失类型 | 含义 |
|---|---|
| 结论损失 | 没有先直接回答问题 |
| 证据损失 | 没有使用项目、课程、论文、竞赛等经历作为支撑 |
| 结构损失 | 回答顺序混乱，缺少清晰主线 |
| 深度损失 | 停留在兴趣、热爱、前景等表层表达 |
| 匹配损失 | 没有连接目标导师、实验室或研究方向 |
| 边界损失 | 没有说清自己真实负责的部分，容易被质疑 |

### 8.5 示例

临场回答：

```text
我对人工智能比较感兴趣，也觉得大模型发展很快，所以希望继续学习。
```

冷静回答：

```text
我之前做过一个大模型辅助论文中译英 TeX 编辑器，在项目中发现大模型虽然生成能力强，但在科研写作场景中容易出现术语不一致、格式破坏和上下文不稳定的问题。因此我希望之后继续研究大模型在科研辅助场景中的可靠性和可控性。
```

诊断输出：

```text
临场主要损失：
1. 证据损失：没有提到论文翻译项目；
2. 深度损失：只说“大模型发展快”，没有说具体问题；
3. 匹配损失：没有说明该问题如何连接到未来研究方向；
4. 结构损失：没有形成“经历 → 问题 → 方向 → 计划”的链条。
```

### 8.6 产品价值

这个角色把“我一紧张就没答好”具体化为可分析的表达损失。

## 9. 角色五：Answer Synthesizer 回答融合重构器

### 9.1 角色定位

回答融合重构器负责根据诊断结果生成更稳的回答版本。

它不是普通润色器，而是需要遵守几个原则：

```text
不编造用户没有提供的经历
不过度包装
保留用户真实材料
适合口述
降低追问风险
体现科研问题意识
```

### 9.2 输入

```text
面试问题
多个回答版本
材料证据
风险审查结果
问题意图
```

### 9.3 输出

```text
30 秒口述版
60 秒完整回答版
更自然版本
更科研表达版本
不建议使用的表达
```

### 9.4 示例

输出：

```text
30 秒口述版：
我对这个方向的兴趣不是单纯来自“AI 很热门”，而是来自我之前做的一个大模型辅助论文中译英 TeX 编辑器。在项目中我发现，大模型在科研写作场景里虽然生成能力很强，但仍然会出现术语不一致、格式破坏和上下文不稳定等问题。所以我希望研究生阶段能继续围绕大模型在科研辅助场景中的可靠性和可控性问题深入学习。
```

### 9.5 产品价值

这个角色把诊断结果转化成用户可以练习的答案，但同时避免把回答改得过度夸张。

## 10. 角色六：Training Planner 训练规划器

### 10.1 角色定位

训练规划器负责把一次复盘转化成下一步训练。

很多 AI 工具只给用户一个改写后的答案，但用户下次遇到类似问题时仍然不会组织。

训练规划器要输出：

```text
下次回答公式
救场模板
下一道建议训练题
下一场面试前准备清单
个人弱点提醒
```

### 10.2 输入

```text
复盘报告
风险点
用户材料
回答重构结果
```

### 10.3 输出

```text
可迁移回答公式
救场句式
下一题推荐
准备清单
弱点标签
```

### 10.4 示例

```text
同类问题回答公式：
经历来源 → 具体问题 → 方向连接 → 未来计划

救场模板：
我对这个方向的兴趣不是泛泛产生的，而是来自 ____。在这个过程中，我发现 ____ 是一个值得继续探索的问题。因此我希望研究生阶段围绕 ____ 继续深入。

下一题建议：
请用 60 秒说明你在这个项目中具体负责了什么，以及哪些部分不是你完成的。
```

### 10.5 产品价值

这个角色让产品从“一次回答点评”升级为“训练闭环”。

---

## 11. “导师组会审”产品化表达

为了让多角色机制更容易被用户和评委理解，界面上不一定要直接说“多 Agent”。可以包装成：

> **导师组会审报告**

其中包含：

1. **问题意图导师**：判断这道题真正考察什么；
2. **材料证据导师**：判断哪些经历应该被说出来；
3. **项目深挖导师**：识别项目和贡献风险；
4. **表达教练**：判断结构和口述自然度；
5. **风险审查员**：检查空泛、夸大和追问风险；
6. **训练规划师**：给出下一步练习建议。

这样的表达比“我们用了多 Agent”更自然，也更贴保研场景。

# 第二部分：结构化复盘系统

## 12. 为什么结构化复盘比单纯多 Agent 更重要

多 Agent 是一种实现和组织方式，但不是产品价值本身。

真正能让产品强起来的是：

```text
材料被结构化
回答被结构化比较
风险被结构化呈现
建议被结构化沉淀
训练被结构化延续
```

因此，除了多角色诊断链，还应该设计一套结构化复盘机制。

这套机制可以包括：

1. Evidence Bank：材料证据库；
2. Rubric Scoring：保研面试评分矩阵；
3. Answer Diff：回答差异分析；
4. Risk Radar：回答风险雷达；
5. Counterfactual Answers：反事实回答比较；
6. Rescue Templates：救场模板库；
7. Replay Cards：复盘卡片；
8. Weakness Profile：个人弱点画像；
9. Adaptive Training：下一题推荐；
10. Authenticity Guard：真实性守门员。

---

## 13. 升级方向一：Evidence Bank 材料证据库

### 13.1 核心思路

用户提供的简历、项目经历、自我介绍和个人陈述，不应该只是作为一整段上下文塞给大模型，而应该先被拆解成“证据卡”。

每张证据卡包含：

```text
材料类型
材料内容
可支撑的问题类型
可证明的能力
可能引发的追问
使用风险
建议表达方式
```

### 13.2 示例证据卡

```text
证据卡：大模型论文中译英 TeX 编辑器

材料类型：项目经历
可支撑问题：
- 为什么选择 AI/NLP 方向？
- 请介绍一个项目经历。
- 你有什么科研兴趣？
- 如何看待大模型应用？

可证明能力：
- 工程实现能力
- Prompt 设计经验
- 对真实科研写作场景的理解
- 从应用问题中提炼研究问题的能力

潜在风险：
- 需要说明为什么不是直接套壳 ChatGPT；
- 需要说明如何评价翻译质量；
- 需要说明个人贡献和 AI Coding 工具贡献的边界。
```

### 13.3 产品价值

材料证据库让产品拥有“个人化基础”。

后续复盘时，系统可以判断：

```text
这道题本来应该调用哪些材料？
用户实际调用了哪些材料？
用户遗漏了哪些关键材料？
用户用了哪些不适合这道题的材料？
```

这比普通 AI 点评更具体。

---

## 14. 升级方向二：材料召回率

### 14.1 核心思路

材料召回率用于衡量：

> 对于当前面试问题，用户回答调用了多少本应使用的个人材料。

### 14.2 示例

问题：

```text
你为什么选择这个研究方向？
```

系统认为应该使用的材料：

```text
1. 大模型论文翻译项目；
2. Prompt 设计经历；
3. 科研写作场景中的术语一致性问题。
```

用户临场回答：

```text
我对人工智能比较感兴趣，也觉得这个方向发展很快。
```

系统输出：

```text
材料召回率：0 / 3

你本来有 3 个很适合支撑这道题的材料，但临场回答没有使用任何一个，因此回答显得像模板表达。
```

### 14.3 产品价值

材料召回率把“你说得太空泛”变成了一个可解释指标。

它能让用户清楚看到：

> 我不是没内容，而是没有把已有内容调出来。

---

## 15. 升级方向三：Rubric Scoring 保研面试评分矩阵

### 15.1 核心思路

不要让 AI 自由点评，而是建立固定评价维度。

### 15.2 推荐评分维度

| 维度 | 含义 |
|---|---|
| 正面回答程度 | 是否直接回应问题 |
| 材料证据使用 | 是否使用自己的项目、论文、课程或经历 |
| 科研问题意识 | 是否说出具体问题，而不是泛泛兴趣 |
| 个人贡献清晰度 | 是否讲清自己具体做了什么 |
| 导师/方向匹配度 | 是否连接目标导师、学院或研究方向 |
| 真实性与边界感 | 是否避免过度包装和夸大贡献 |
| 追问抗压性 | 当前回答是否经得起下一轮追问 |
| 口述自然度 | 是否像真实面试中能自然说出来的话 |

### 15.3 每个维度输出内容

```text
评分
原因
具体问题
缺失信息
改进建议
```

### 15.4 产品价值

评分矩阵让复盘从“主观点评”变成“基于保研复试标准的结构化诊断”。

---

## 16. 升级方向四：Answer Diff 回答差异分析

### 16.1 核心思路

用户常常会提供多个回答版本。产品不应该只判断哪个更好，而应该解释：

```text
这个版本比另一个版本多了什么？
少了什么？
新增的信息是否有效？
新增的信息是否带来风险？
哪个版本的结构更适合口述？
```

### 16.2 差异类型

| 差异类型 | 含义 |
|---|---|
| 新增亮点 | 后一个回答加入了有效信息 |
| 遗漏信息 | 某个回答缺少关键内容 |
| 风险新增 | 某个回答引入了难以承接的新概念 |
| 结构优化 | 回答顺序更清晰 |
| 表达降级 | 听起来更长，但更空泛或更危险 |

### 16.3 示例

```text
回答 B 相比回答 A 新增了“项目经历”，这是有效补充。

但回答 B 新增了“可靠人机协作”这一概念。如果用户没有读过相关论文，可能会引发导师追问。因此建议保留项目经历，但将“可靠人机协作”改为更贴近用户材料的“科研写作场景中的可控性问题”。
```

### 16.4 产品价值

Answer Diff 抓住了用户真实心理：

> 我事后想到的版本是不是真的更好？

---

## 17. 升级方向五：Risk Radar 回答风险雷达

### 17.1 核心思路

每次复盘后，系统给出一个风险面板，让用户快速看到当前回答的主要风险。

### 17.2 推荐风险类型

```text
空泛表达风险
证据不足风险
贡献不清风险
过度包装风险
导师匹配不足
研究问题不具体
追问承接不足
口述不自然风险
```

### 17.3 示例输出

```text
风险雷达：
- 空泛表达风险：高
- 证据不足风险：高
- 贡献不清风险：中
- 过度包装风险：低
- 导师匹配不足：中
- 追问承接不足：高

最大风险：
你说“我对人工智能很感兴趣”，但没有说明兴趣来源，也没有用项目经历支撑。
```

### 17.4 产品价值

风险雷达能把长篇报告压缩为清晰的诊断面板，适合 Demo 展示。

---

## 18. 升级方向六：Counterfactual Answers 反事实回答比较

### 18.1 核心思路

除了比较用户提供的回答，系统还可以生成不同策略的反事实版本，并解释它们的优劣。

例如：

```text
保守稳妥版
进取亮点版
真实口语版
科研表达版
```

### 18.2 示例

问题：

```text
你为什么选择这个方向？
```

保守稳妥版：

```text
我本科阶段通过课程和项目接触到大模型应用，对这个方向产生了兴趣，希望研究生阶段继续系统学习。
```

进取亮点版：

```text
我希望研究大模型在科研写作场景中的可靠性和可控性问题。
```

AI 分析：

```text
保守版更安全，但亮点不足；
进取版更有研究意识，但需要用户准备具体项目证据和相关论文；
推荐采用稳健融合版：先说项目经历，再引出可靠性问题，不直接夸大为成熟研究计划。
```

### 18.3 产品价值

这个功能解决了保研学生常见困惑：

> 我应该说得保守一点，还是包装得强一点？

---

## 19. 升级方向七：Rescue Templates 救场模板库

### 19.1 核心思路

很多学生不是完全不会答，而是临场卡住。系统应根据复盘结果生成可套用的救场句式。

### 19.2 常见救场模板

#### 兴趣空泛时

```text
我对这个方向的兴趣不是泛泛产生的，而是来自一次具体经历：____。
在这个过程中，我发现 ____ 是一个值得进一步探索的问题。
所以我希望研究生阶段能围绕 ____ 继续深入。
```

#### 个人贡献被追问时

```text
这个项目中我主要负责的是 ____。
团队其他成员主要负责 ____。
我自己做得比较深入的部分是 ____。
如果继续改进，我认为可以从 ____ 入手。
```

#### 导师方向匹配不足时

```text
我了解到老师团队关注 ____。
我之前的经历虽然还比较初步，但和其中的 ____ 有一定关联。
所以我希望后续能先从 ____ 开始补充，再逐步深入到 ____。
```

#### 被质疑项目创新性时

```text
这个项目目前更多是应用探索，严格来说创新性还有限。
但它让我意识到 ____ 这个问题在真实场景中仍然存在。
如果继续做，我会优先从 ____ 和 ____ 两方面加强。
```

### 19.3 产品价值

救场模板比“标准答案”更适合训练，因为用户可以迁移到不同问题。

---

## 20. 升级方向八：Replay Card 复盘卡片

### 20.1 核心思路

长篇复盘报告可能很有用，但用户不一定会反复看。每次复盘后，系统应生成一张短卡片，方便用户保存、复制和下次复习。

### 20.2 卡片内容

```text
问题：你为什么选择这个方向？

原回答问题：
空泛兴趣，没有项目证据。

本次最大改进点：
把“我感兴趣”改成“我的兴趣来自某个项目中遇到的具体问题”。

下次回答公式：
经历 → 问题 → 方向 → 计划

救场句：
我对这个方向的兴趣来自于我在 ____ 项目中遇到的 ____ 问题。

下一次准备：
1. 补充一个项目指标；
2. 准备一篇相关论文；
3. 明确个人贡献边界。
```

### 20.3 产品价值

复盘卡片把一次复杂分析沉淀为可复习、可迁移的小单元。

---

## 21. 升级方向九：Weakness Profile 个人弱点画像

### 21.1 核心思路

如果用户复盘多次，系统可以总结用户的常见问题。

即使不做账号系统，也可以用本地历史记录实现简单版本。

### 21.2 示例输出

```text
最近 5 次复盘中，你最常见的问题：

1. 证据损失：4 次
2. 回答开头空泛：3 次
3. 个人贡献不清：3 次
4. 导师方向连接不足：2 次

你的主要问题不是没有经历，而是临场时没有主动调用经历。
建议下一阶段重点练习：
“结论 → 项目证据 → 研究问题 → 未来计划”的回答结构。
```

### 21.3 产品价值

个人弱点画像让产品从一次性工具升级成长期训练系统。

---

## 22. 升级方向十：Adaptive Training 下一题推荐

### 22.1 核心思路

不要做传统追问树，而是根据复盘暴露的问题推荐下一道最该练的题。

### 22.2 示例

如果本次问题是“个人贡献不清”，系统推荐：

```text
下一题建议：
请用 60 秒说明你在这个项目中具体负责了什么，以及哪些部分不是你完成的。
```

如果本次问题是“导师匹配不足”，系统推荐：

```text
下一题建议：
请说明你的项目经历如何连接到目标导师的研究方向。
```

如果本次问题是“科研问题意识不足”，系统推荐：

```text
下一题建议：
你的项目里最值得继续研究的问题是什么？为什么？
```

### 22.3 产品价值

这不是普通连续追问，而是基于复盘结果的个性化训练路径。

---

## 23. 升级方向十一：Authenticity Guard 真实性守门员

### 23.1 核心思路

保研面试中，过度包装非常危险。系统应该专门检查回答是否夸大、不真实或难以承接。

### 23.2 检查内容

```text
是否夸大个人贡献
是否使用不熟悉术语
是否把兴趣包装成成熟研究计划
是否把课程作业包装成科研成果
是否暗示读过其实没读过的论文
是否说了无法承接的未来方向
```

### 23.3 示例

用户回答：

```text
我希望继续研究多模态智能体。
```

但用户材料中没有任何多模态经历。

系统输出：

```text
真实性风险：
你提到“多模态智能体”，但当前材料中没有任何多模态相关经历。如果这样表达，导师可能追问你读过哪些相关论文、做过哪些相关项目。

建议改为：
我目前更关注大模型在科研写作场景中的可靠性问题，未来也希望逐步了解多模态智能体等更复杂的人机协作方向。
```

### 23.4 产品价值

这个机制能让产品区别于普通“帮我包装得更好”的工具。

---

## 24. 升级方向十二：题型识别与回答公式匹配

### 24.1 核心思路

系统先识别当前问题属于哪类，再匹配不同的回答公式和评分标准。

### 24.2 常见题型

```text
动机类：为什么选择方向/实验室
项目类：介绍项目/贡献/难点
科研潜力类：未来计划/研究兴趣
压力质疑类：你这个项目有什么创新
专业基础类：解释概念/方法原理
行为经历类：困难、合作、失败
```

### 24.3 回答公式示例

#### 项目类

```text
项目背景 → 问题定义 → 我的贡献 → 方法选择 → 结果验证 → 局限反思
```

#### 动机类

```text
经历来源 → 具体问题 → 方向连接 → 未来计划
```

#### 压力质疑类

```text
先承认边界 → 说明已有工作 → 给出证据 → 提出改进方向
```

#### 科研潜力类

```text
已有经历 → 发现问题 → 初步理解 → 未来计划 → 当前不足
```

### 24.4 产品价值

题型识别让产品的反馈更稳定，也便于用户学习一类问题的通用结构。

---

## 25. 升级方向十三：可信度与信息缺口提示

### 25.1 核心思路

AI 不应该装作全知。如果用户提供的信息不足，系统应明确提示分析可信度和信息缺口。

### 25.2 示例

```text
分析可信度：中

原因：
你没有提供目标导师方向，因此关于“导师匹配度”的分析只能基于一般保研面试标准。
如果补充导师研究方向，系统可以进一步判断哪些表达更适合。
```

或者：

```text
信息缺口：
1. 你提到“效果提升明显”，但没有给出指标；
2. 你提到“读过相关论文”，但没有说明论文名称；
3. 你提到“负责核心模块”，但没有说明具体工作。
```

### 25.3 产品价值

可信度提示能降低 AI 胡说风险，也体现产品专业性。

---

# 第三部分：组合后的升级版产品形态

## 26. 升级后的产品定位

升级后，Interview Replay 可以定义为：

> **一个面向保研学生的结构化 AI 面试复盘引擎。它不是简单模拟面试，而是先理解用户材料，再从导师视角、材料视角和表达视角诊断回答，最后将一次回答复盘沉淀为下一次可复用的训练策略。**

相比基础版本，升级版的关键词是：

```text
材料证据库
临场差距诊断
回答差异分析
导师风险雷达
真实性守门员
救场模板
复盘卡片
个性化训练推荐
```

---

## 27. 推荐的三层系统设计

### 27.1 第一层：材料理解层

目标：

```text
把用户材料转化为证据卡、风险点和可用表达素材。
```

主要能力：

```text
材料证据库
证据卡生成
材料适用场景识别
潜在追问识别
真实性风险预警
```

输出：

```text
项目证据
贡献证据
方向证据
风险术语
潜在追问
可用回答素材
```

---

### 27.2 第二层：复盘诊断层

目标：

```text
根据面试问题和回答进行多视角诊断。
```

主要能力：

```text
问题意图识别
材料召回率分析
临场损失分析
多版本回答对比
逐句诊断
风险雷达
真实性检查
```

输出：

```text
问题意图
回答评分矩阵
材料遗漏
临场损失
追问风险
过度包装风险
回答差异报告
```

---

### 27.3 第三层：训练生成层

目标：

```text
把诊断结果转化为下一步可执行训练。
```

主要能力：

```text
最佳融合回答
30 秒/60 秒口述版
救场模板
可迁移回答公式
下一题推荐
复盘卡片
个人弱点画像
```

输出：

```text
改进版回答
回答公式
救场句式
下一场准备清单
下一题训练建议
复盘卡片
```

## 28. 升级后的两种模式

## 28.1 面试前模拟模式

升级后流程：

```text
输入背景材料
↓
生成材料证据库
↓
输入或生成面试问题
↓
限时临场回答
↓
冷静复盘回答
↓
问题意图识别
↓
材料召回率分析
↓
临场损失诊断
↓
导师风险雷达
↓
最佳融合回答
↓
救场模板 + 下一题推荐
```

核心输出：

```text
你紧张时丢掉了什么
你本来有哪些材料可以用
你的回答最大风险是什么
下次如何用模板救场
下一道应该练什么
```

---

## 28.2 面试后复盘模式

升级后流程：

```text
输入真实面试问题
↓
输入面试背景和个人材料
↓
输入多个回答版本
↓
问题意图识别
↓
多版本 Answer Diff
↓
评分矩阵比较
↓
真实性守门员检查
↓
导师追问风险预测
↓
最佳融合回答
↓
复盘卡片 + 可迁移回答公式
```

核心输出：

```text
哪个回答更稳
为什么更稳
哪个版本引入了新风险
哪些材料没有用上
最终推荐怎么答
这类问题以后怎么答
```

# 第四部分：16 小时内的取舍建议

## 29. 不要全部做

虽然上面列出了很多升级方向，但 16 小时项目挑战不能贪多。

最重要的是做出一个：

```text
可运行
可展示
可解释
有明显产品差异
能体现用户价值
```

的最小闭环。

因此，需要分优先级。

---

## 30. P0：真实轻量多 Agent 诊断链

P0 的目标是把现有“一次 AI 生成复盘报告”升级为“多角色 Agent 串联诊断”。这里的多 Agent 不是引入复杂外部框架，而是在项目内实现一个轻量 Agent 编排框架：每个角色有独立 prompt、独立结构化输出、独立 LLM 调用，最后由报告聚合器整合为前端报告。

因此，P0 不是“单次 LLM 调用里模拟多角色”，而是至少包含以下真实 Agent 节点：

```text
Material Analyst Agent        材料分析器
Question Intent Agent         问题意图分析器
Evidence Mapper Agent         材料证据匹配器
Skeptical Professor Agent     导师风险审查员
Gap / Diff Diagnoser Agent    临场差距 / 多版本差异诊断器
Answer Synthesizer Agent      回答融合重构器
Training Planner Agent        训练规划器
Report Composer               最终报告生成器
```

P0 的工程重点是做出可运行、可解释、可 Demo 的多 Agent 闭环，而不是追求复杂调度、记忆系统或外部 Agent 框架。

### 30.1 P0 Agent 编排流程

面试前模拟模式：

```text
用户输入：背景材料 + 面试问题 + 临场回答 + 冷静回答
↓
Material Analyst Agent：提取材料证据库
↓
Question Intent Agent：判断问题真实意图
↓
Evidence Mapper Agent：计算材料召回率和遗漏材料
↓
Skeptical Professor Agent：识别导师追问风险、过度包装和真实性风险
↓
Gap Diagnoser Agent：诊断临场回答相对冷静回答丢失了什么
↓
Answer Synthesizer Agent：生成最佳融合回答
↓
Training Planner Agent：生成救场模板、复盘卡片和下一题建议
↓
Report Composer：整合为 PreReplayReport
```

面试后复盘模式：

```text
用户输入：背景材料 + 面试问题 + 多个回答版本
↓
Material Analyst Agent：提取材料证据库
↓
Question Intent Agent：判断问题真实意图
↓
Evidence Mapper Agent：判断各版本材料调用情况
↓
Skeptical Professor Agent：识别导师追问风险、真实性风险和过度包装
↓
Answer Diff Diagnoser Agent：比较多个回答版本的差异
↓
Answer Synthesizer Agent：生成最佳融合回答
↓
Training Planner Agent：生成可迁移公式、复盘卡片和下一题建议
↓
Report Composer：整合为 PostReplayReport
```

### 30.2 P0 必须实现的 Agent 输出

P0 不要求每个 Agent 都输出长篇报告，但必须有清晰结构化结果。

#### Material Analyst Agent

输出：

```text
evidenceCards
```

每张证据卡包含：

```text
材料名称
材料类型
可证明的能力
适合支撑的问题类型
可能引发的追问
使用风险
建议表达
```

#### Question Intent Agent

输出：

```text
questionIntent
evaluationFocus
idealAnswerLayers
```

#### Evidence Mapper Agent

输出：

```text
materialRecall
missingEvidence
usedEvidence
```

核心指标：

```text
材料召回率 usedCount / expectedCount
```

#### Skeptical Professor Agent

输出：

```text
riskRadar
followUpRisks
authenticityWarnings
```

P0 固定风险维度：

```text
空泛表达风险
证据不足风险
贡献不清风险
过度包装风险
导师匹配不足风险
追问承接不足风险
```

#### Gap / Diff Diagnoser Agent

面试前输出：

```text
liveLossAnalysis
liveAnswerDiagnosis
calmAnswerImprovements
```

面试后输出：

```text
answerRanking
versionReviews
sentenceDiagnosis
```

#### Answer Synthesizer Agent

输出：

```text
bestMergedAnswer
```

要求：

```text
不编造用户没有提供的经历
不过度包装
保留真实材料
适合口述
降低追问风险
```

#### Training Planner Agent

面试前输出：

```text
rescueTemplate
nextPracticeAdvice
replayCard
```

面试后输出：

```text
transferableFormula
nextInterviewChecklist
replayCard
```

### 30.3 P0 技术实现边界

P0 使用“项目内轻量多 Agent 编排框架”，不使用外部复杂框架。

允许：

```text
多个服务端 LLM 调用
每个 Agent 独立 prompt
Agent 输出结构化 JSON
服务端顺序编排
局部失败时给清晰错误或降级
前端展示 Agent 结果整合后的报告
```

不做：

```text
LangChain / CrewAI 等重框架
数据库
长期记忆
工具调用系统
RAG 知识库
院校/导师爬虫
Agent 自主规划循环
无限重试
```

建议代码结构：

```text
src/lib/agents/
  types.ts
  runner.ts
  material-agent.ts
  intent-agent.ts
  evidence-agent.ts
  professor-agent.ts
  gap-agent.ts
  diff-agent.ts
  synthesizer-agent.ts
  training-agent.ts
  composer.ts
```

### 30.4 P0 Demo 必须展示的模块

前端报告必须展示：

```text
材料证据库
材料召回率
风险雷达
```

### 30.5 P0 验收标准

```text

```

## 31. P1：多 Agent 增强与可解释性

P1 在 P0 的轻量多 Agent 链稳定之后做增强。目标不是继续堆报告字段，而是让多 Agent 更稳定、更可解释、更适合 Demo。

### 31.1 Agent 执行轨迹展示

前端展示每个角色的简短结论：

```text
材料分析器：你有 3 张可用证据卡
问题意图分析器：本题主要考察科研动机和方向匹配
证据匹配器：材料召回率 1/3
导师风险审查员：证据不足风险高
训练规划器：建议下一题练个人贡献边界
```

这能让评委直观看到产品不是一次 Prompt 套壳。

### 31.2 Agent 并行与耗时优化

P0 可以顺序执行，P1 可以优化为部分并行：

```text
Material Analyst 完成后
Question Intent / Evidence Mapper / Skeptical Professor 可部分并行
Gap/Diff Diagnoser 与 Professor 可并行
Synthesizer 和 Training Planner 等待前序结果
Composer 最后执行
```

目标是降低等待时间，但不要为了并行破坏可维护性。

### 31.3 Agent 输出容错和降级

增强：

```text
单个 Agent 返回 JSON 异常时可重试一次
非核心 Agent 失败时报告中显示“该角色分析失败”
核心 Agent 失败时返回清晰错误
记录每个 Agent 的错误码和耗时
```

为每个 Agent prompt 标注版本：

```text
material-agent@v1
intent-agent@v1
professor-agent@v1
```

便于后续调试和交接。

## 32. P2：暂缓，不作为赛时重点

P2 中很多能力需要数据库、长期状态或更复杂的数据模型，不适合在当前比赛阶段做。

### 32.1 个人弱点画像

需要保存多次复盘记录并统计趋势，通常依赖：

```text
账号或本地历史
报告持久化
弱点标签归一化
跨报告统计逻辑
```

当前暂缓。

### 32.2 反事实回答比较

生成保守版、进取版、科研表达版等不同策略回答有价值，但会增加报告长度和模型输出不稳定性。

当前暂缓。

### 32.3 可信度提示

可信度提示有助于降低 AI 胡说风险，但需要更系统地判断输入完整度和不同分析模块的证据来源。

当前暂缓；如果 P0 做得很快，可以只在 Prompt 中保留“信息不足要说明”的通用要求。

### 32.4 题型识别与公式匹配

自动识别题型并匹配公式需要维护题型规则和模板体系。

当前暂缓；P0/P1 中只让模型在可迁移回答公式里自然体现。

### 32.5 长期训练记录和个性化推荐

这类功能依赖数据持久化，不适合当前无数据库的赛时架构。

当前暂缓。

# 第五部分：Demo 中如何表达升级价值

## 34. Demo 讲法

可以这样介绍产品升级点：

> 我们没有把大模型简单用于“生成面试题”或“润色回答”。我们把保研面试复盘拆成了三个层次：第一，系统先理解用户材料，提取可以支撑回答的证据；第二，系统对临场回答和复盘回答进行结构化诊断，判断材料有没有用上、紧张时丢失了什么、回答会不会引发导师追问；第三，系统把诊断结果转化成最佳融合回答、救场模板和下一题训练建议。

这段话能够体现：

```text
不是套壳 ChatGPT
不是普通模拟面试
不是普通润色工具
而是结构化复盘系统
```

# 第六部分：升级后的项目亮点总结

## 36. 亮点一：材料驱动，而不是问题驱动

普通面试工具通常是 AI 问问题。  
本项目先理解用户材料，再判断回答有没有用好材料。

核心概念：

```text
材料证据库
材料召回率
材料证据遗漏
```

---

## 37. 亮点二：复盘驱动，而不是模拟驱动

产品不追求完全模拟面试气氛，而是抓住面试中最可训练的部分：回答之后的复盘。

核心概念：

```text
临场回答
冷静回答
临场损失
Answer Diff
```

---

## 38. 亮点三：导师风险审查，而不是单纯润色

产品不会把回答盲目改得更高级，而是提醒用户哪些表达可能引发追问。

核心概念：

```text
风险雷达
真实性守门员
过度包装风险
追问承接风险
```

---

## 39. 亮点四：训练闭环，而不是一次性输出

产品最终不只给出一个更好答案，还给出下一次训练策略。

核心概念：

```text
救场模板
可迁移回答公式
复盘卡片
下一题推荐
个人弱点画像
```

---

## 40. 亮点五：轻量多角色诊断，而不是复杂炫技

项目可以借鉴多 Agent 思路，但不为炫技而炫技。

核心表达：

```text
问题意图分析器
材料证据匹配器
导师风险审查员
临场差距诊断器
回答融合重构器
训练规划器
```

这些角色共同服务于一个目标：

> 让用户知道自己为什么答不好，以及下一次如何答得更稳。

# 第七部分：Product Memo 可直接使用版本

## 41. 升级设计说明

本项目最初的核心是“保研面试复盘”：用户可以在面试前进行限时临场回答并冷静重答，也可以在面试后输入真实问题和多个回答版本，由 AI 分析临场差距和多版本差异。进一步思考后，我们认为如果只是让大模型直接生成点评报告，产品仍然容易停留在 Prompt 工程层面。因此，我们将项目升级为一个结构化复盘系统。

升级后的系统首先会理解用户的个人材料，将简历、项目经历、自我介绍等内容拆解为“证据卡”，判断这些材料分别适合支撑哪些面试问题、可能引发哪些追问。之后，系统围绕某一道面试问题进行诊断：识别问题真实意图，判断用户回答是否调用了合适材料，比较临场回答和冷静回答的差距，分析多个回答版本之间的优劣，并通过风险雷达提示空泛表达、证据不足、贡献不清和过度包装等问题。最后，系统将诊断结果转化为最佳融合回答、救场模板、可迁移回答公式和下一题训练建议。

在 AI 能力设计上，我们没有把大模型主要用于生成面试题，而是将其用于多个更具体的复盘任务：问题意图分析、材料证据匹配、导师风险审查、临场差距诊断、回答融合重构和训练建议生成。这样的轻量多角色设计让产品不只是一次 Prompt 调用，而是一个面向保研场景的回答诊断流程。

相比直接使用 ChatGPT，本产品的差异在于：它提供固定的复盘流程、保研场景下的评价维度、材料召回率和风险雷达等结构化指标，并将一次回答复盘沉淀为下一次可用的训练策略。产品最终希望解决的不是“让 AI 问更多问题”，而是帮助学生清楚知道：我漏掉了什么、为什么漏掉、哪些表达有风险、下次该怎么补上。

---

## 42. 最终一句话总结

> **Interview Replay 的升级方向不是做一个更会聊天的 AI 面试官，而是做一个结构化保研面试复盘引擎：先把用户材料变成证据，再把回答差距变成诊断，最后把诊断结果变成下一次可复用的训练策略。**
