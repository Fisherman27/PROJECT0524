# Interview Replay 质量优化方案

> 文件名：质量优化的英文.md  
> 适用项目：Interview Replay / 研答 Replay  
> 主题：在功能和交互基本完整后，如何提高多 Agent 复盘系统的判断质量、协作质量与输出稳定性。

---

## 1. 质量优化的核心目标

当前产品已经具备较完整的功能闭环：

1. **面试前模拟**：用户进行临场回答与冷静回答，系统分析临场差距；
2. **面试后复盘**：用户输入真实问题和多个回答版本，系统比较优劣；
3. **结构化报告**：输出材料召回率、临场损失、风险雷达、最佳融合回答、救场模板等结果。

在这个阶段，继续增加功能数量并不是最优方向。更重要的是提高系统的判断质量，让产品从：

> 用户输入回答 → AI 生成一份看似合理的点评

升级为：

> 用户输入背景材料和回答 → 系统基于证据进行结构化诊断 → 发现具体风险和缺口 → 输出真实、可承接、可训练的改进策略。

因此，质量优化的核心目标不是“让 Agent 更多”，而是：

> **让每个 Agent 少说漂亮话，多做可验证判断。**

具体来说，系统需要做到：

- 每个 Agent 有清晰职责边界；
- 每个关键结论尽量绑定用户材料或信息缺口；
- 不同 Agent 之间避免重复评价；
- 最终回答不追求华丽，而追求真实、具体、可承接；
- 报告内容分层展示，避免用户被大量 Agent 输出淹没；
- 通过少量高质量样例持续校准 Prompt 和输出格式。

---

## 2. 质量问题的主要来源

### 2.1 Agent 职责重叠

如果各 Agent 边界不清，容易出现重复输出。例如：

- Evidence Mapper 会说材料不足；
- Professor 也会说材料不足；
- Gap Diagnoser 也会说证据缺失；
- Training Planner 又重复一遍建议。

结果是报告看起来很长，但信息密度不高。

### 2.2 AI 过度补全

大模型很容易为了让回答更完整而主动补充用户没有提供的信息，尤其是在保研面试场景中，这非常危险。

例如：

- 用户只说做过 Prompt 设计，AI 却写成“深入研究大模型推理机制”；
- 用户只说参与项目，AI 却写成“主导项目”；
- 用户没有提供实验指标，AI 却写“效果显著提升”。

这种输出会让回答变得漂亮，但被导师追问时很容易穿帮。

### 2.3 缺少证据链

如果系统说：

> 你的回答缺少项目证据。

但没有说明这个判断来自哪里，用户会觉得这是泛泛而谈。

更好的输出应该是：

> 依据：你在背景材料中提到“大模型论文中译英 TeX 编辑器”，但临场回答完全没有使用这个项目来解释研究兴趣。

### 2.4 最终答案过度润色

Synthesizer 如果没有约束，很容易生成一种“完美但危险”的回答。这类回答看起来像标准文书，但不一定适合口述，也不一定能承接追问。

保研面试中的好回答不一定最华丽，而应该：

- 真实；
- 具体；
- 有边界感；
- 能经受追问；
- 听起来像学生本人能自然说出口的话。

### 2.5 输出信息过载

如果把每个 Agent 的完整分析都展示给用户，报告会变得冗长，用户很难抓住重点。

产品需要区分：

- 用户第一眼需要看到什么；
- 用户愿意展开看什么；
- 哪些内容只适合内部调试。

## 3. 总体优化原则

### 3.1 判断优先于生成

Agent 的核心能力不是“写得好”，而是“判断得准”。

每个 Agent 都应该有一个明确的判断问题：

- 这道题到底考什么？
- 这题本来应该调用哪些材料？
- 用户实际用了哪些材料？
- 哪句话会被导师追问？
- 哪些信息是用户临场丢失的？
- 最终回答有没有编造或夸大？

### 3.2 证据优先于观点

所有关键结论都应该尽量满足：

> **有材料依据则引用材料依据；没有材料依据则标注信息缺口。**

例如：

```text
结论：这段回答存在贡献不清风险。
依据：用户回答中说“我负责模型优化”，但背景材料只提到 Prompt 设计，没有说明是否涉及模型训练。
信息缺口：需要补充用户到底优化的是 Prompt、流程、交互，还是模型参数。
```

### 3.3 安全优先于高级感

最终回答的生成应遵循以下优先级：

```text
真实性 > 可承接性 > 具体性 > 结构清晰 > 口述自然 > 高级感
```

如果一个表达很高级，但用户无法解释，就应该降级为更稳妥的表达。

### 3.4 分层展示优先于完整堆叠

内部可以有多个 Agent 分析，但用户界面不应该把所有输出原封不动展示出来。最终报告应该经过整理、聚合和优先级排序。

### 3.5 信息缺口优先于强行判断

当材料不足时，系统应该敢于说：

> 当前无法可靠判断，需要补充信息。

这比强行给结论更专业。

## 4. Agent 职责边界设计

质量优化第一步，是明确每个 Agent 的职责边界。Agent 越多，越需要边界清晰。

### 4.1 Material Analyst：材料分析员

#### 核心任务

只负责从用户输入的背景材料中提取可用证据，不评价回答好坏。

#### 输入

- 简历材料；
- 项目经历；
- 科研经历；
- 竞赛经历；
- 课程背景；
- 自我介绍；
- 个人陈述；
- 未来研究兴趣。

#### 输出

- 证据卡；
- 可支撑的问题类型；
- 潜在风险；
- 信息缺口。

#### 不应该做

- 不评价用户回答；
- 不生成最终答案；
- 不判断哪个回答更好；
- 不润色材料。

#### 输出示例

```text
证据卡 #1：大模型论文中译英 TeX 编辑器
类型：项目经历
可支撑问题：
- 为什么选择 AI/NLP 方向？
- 请介绍一个项目经历；
- 你为什么对科研写作场景感兴趣？

可支撑点：
- 用户接触过真实科研写作场景；
- 用户有 Prompt 设计和工程实现经验；
- 用户能从应用场景中提炼问题。

风险：
- 需要解释为什么不是简单调用 ChatGPT；
- 需要说明翻译质量如何评价；
- 需要说明个人贡献边界。
```

---

### 4.2 Question Intent Analyst：问题意图分析员

#### 核心任务

只负责判断面试问题背后的真实考察意图。

#### 输入

- 面试问题；
- 面试类型；
- 目标方向；
- 可选的院校或导师背景。

#### 输出

- 问题类型；
- 真实考察目标；
- 回答应覆盖的层次；
- 常见答偏方式。

#### 不应该做

- 不评价用户回答；
- 不生成标准答案；
- 不分析用户材料；
- 不输出训练建议。

#### 输出示例

```text
问题：你为什么选择我们实验室？

问题类型：动机类 / 导师匹配类

真实考察目标：
1. 用户是否了解实验室方向；
2. 用户的过往经历是否和该方向匹配；
3. 用户是否有真实研究兴趣，而不是海投模板；
4. 用户是否能说出未来想继续探索的问题。

常见答偏：
- 只说“感兴趣”；
- 只说“老师方向很好”；
- 没有连接个人经历；
- 没有说明具体研究问题。
```

---

### 4.3 Evidence Planner：证据规划员

#### 核心任务

在看用户回答之前，先判断这道题本来应该调用哪些材料。

#### 输入

- 面试问题；
- 问题意图；
- 用户证据卡。

#### 输出

- 应调用证据列表；
- 每条证据的作用；
- 不建议调用的材料；
- 需要补充的信息。

#### 为什么要单独拆出来

“应该说什么”和“实际说了什么”必须分开。  
如果看完用户回答再判断遗漏，模型容易被回答带偏。

#### 输出示例

```text
这道题建议调用的证据：

1. 证据卡 #1：论文中译英 TeX 编辑器
作用：说明用户兴趣来自真实项目，而不是泛泛兴趣。

2. 证据卡 #2：Prompt 设计经历
作用：说明用户实际接触过大模型应用中的可控性问题。

3. 证据卡 #3：术语一致性问题
作用：把项目经历提升为一个可研究的问题意识。

不建议调用：
- 与目标方向无关的课程成绩；
- 无法展开说明的泛泛竞赛经历。
```

---

### 4.4 Evidence Mapper：证据映射员

#### 核心任务

只负责判断用户回答实际用了哪些材料，并与 Evidence Planner 的结果对比。

#### 输入

- 应调用证据列表；
- 用户回答；
- 用户背景材料。

#### 输出

- 实际使用的证据；
- 遗漏的证据；
- 材料召回率；
- 证据使用质量。

#### 不应该做

- 不生成最终回答；
- 不做综合评分；
- 不进行导师追问风险分析；
- 不写训练计划。

#### 输出示例

```text
应使用证据：3 条
实际使用证据：0 条

材料召回率：0 / 3

遗漏证据：
1. 没有提到论文中译英 TeX 编辑器项目；
2. 没有提到 Prompt 设计经历；
3. 没有提到科研写作中的术语一致性问题。

诊断：
用户临场回答只表达了“对 AI 感兴趣”，没有调用任何个人材料，因此回答显得像模板表达。
```

---

### 4.5 Gap Diagnoser / Diff Analyst：差距诊断员

#### 核心任务

在面试前模拟模式中，比较临场回答和冷静回答之间的差距。

#### 输入

- 临场回答；
- 冷静回答；
- 应调用证据；
- 实际使用证据。

#### 输出

- 临场损失类型；
- 冷静回答新增信息；
- 仍然缺失的信息；
- 临场问题的根因判断。

#### 不应该做

- 不生成最终答案；
- 不做导师风险审查；
- 不做材料提取。

#### 可识别的临场损失类型

| 损失类型 | 说明 |
|---|---|
| 结论损失 | 没有直接回答问题 |
| 证据损失 | 没有使用项目、论文、课程等材料 |
| 结构损失 | 回答顺序混乱 |
| 深度损失 | 停留在泛泛兴趣和热情 |
| 匹配损失 | 没有连接目标方向 |
| 边界损失 | 没有说清个人贡献边界 |

#### 输出示例

```text
临场损失：

1. 证据损失
临场回答没有使用项目经历；冷静回答补充了论文中译英 TeX 编辑器项目。

2. 深度损失
临场回答只说“大模型发展很快”；冷静回答补充了“术语一致性、格式破坏、上下文不稳定”等具体问题。

3. 方向损失
临场回答没有说明未来想研究什么；冷静回答初步连接到“大模型在科研辅助场景中的可靠性和可控性”。
```

---

### 4.6 Skeptical Professor：导师风险审查员

#### 核心任务

站在保研导师视角，找出回答中最可能被追问或质疑的地方。

#### 输入

- 用户回答；
- 背景材料；
- 问题意图；
- 证据卡；
- 信息缺口。

#### 输出

每个风险点应包含：

1. 风险表述；
2. 导师可能怎么追问；
3. 为什么危险；
4. 用户当前能不能接住；
5. 安全回应建议。

#### 不应该做

- 不负责鼓励；
- 不负责润色；
- 不生成完整最佳回答；
- 不输出泛泛建议。

#### 输出示例

```text
风险表述：我主要负责模型优化。

导师追问：
你优化的是模型参数、Prompt 模板，还是推理流程？

危险原因：
如果用户实际只做了 Prompt 设计，却说“模型优化”，容易让导师误以为用户参与了模型训练或算法改进，存在贡献夸大风险。

当前承接能力：
中低。材料中只提到 Prompt 设计，没有提供模型训练或参数调整相关经历。

安全回应：
我这里的优化主要指 Prompt 模板、输出格式约束和交互流程优化，不涉及模型参数训练。
```

### 4.7 Answer Synthesizer：安全回答生成器

#### 核心任务

根据诊断结果生成真实、具体、可承接、适合口述的回答。

#### 输入

- 问题意图；
- 用户材料；
- 应调用证据；
- 用户原回答；
- 差距诊断；
- Professor 风险点；
- 信息缺口。

#### 输出

- 30 秒版回答；
- 60 秒版回答；
- 可选的更口语版；
- 可选的更科研表达版；
- 使用的证据说明；
- 降低风险的表达说明。

#### 生成原则

```text
真实性 > 可承接性 > 具体性 > 结构清晰 > 口述自然 > 高级感
```

#### 禁止项

- 不能编造用户没提供的经历；
- 不能把参与说成主导；
- 不能把工具使用包装成个人能力；
- 不能默认用户读过某篇论文；
- 不能使用用户无法承接的高级术语；
- 不能写“显著提升”“深入研究”等没有证据的表达；
- 不能把团队成果说成个人贡献。

#### 输出示例

```text
30 秒安全回答：

我对这个方向的兴趣不是单纯因为大模型很热，而是来自我之前做过的一个论文中译英 TeX 编辑器项目。在这个项目中，我发现大模型在科研写作场景里虽然生成能力很强，但仍然容易出现术语不一致、格式被破坏和上下文不稳定的问题。因此我希望研究生阶段能进一步学习大模型在科研辅助场景中的可靠性和可控性问题。

使用证据：
- 证据卡 #1：论文中译英 TeX 编辑器项目；
- 证据卡 #2：Prompt 设计经历；
- 证据卡 #3：科研写作中的术语一致性问题。

风险控制：
没有使用“深入研究”“显著提升”等缺少证据的表达，也没有夸大为模型训练或算法创新。
```

### 4.8 Verifier / Critic：最终校验员

#### 核心任务

检查 Synthesizer 生成的最终回答是否真实、安全、可承接。

#### 输入

- 最终回答；
- 用户材料；
- 证据卡；
- Professor 风险点；
- 问题意图。

#### 输出

- 是否通过；
- 存在的问题；
- 需要修改的句子；
- 修改建议。

#### 检查项

```text
1. 是否使用了用户没有提供的信息？
2. 是否夸大了用户贡献？
3. 是否引入用户无法承接的术语？
4. 是否正面回答了面试问题？
5. 是否保留了足够具体的证据？
6. 是否适合口述？
7. 是否存在新的追问风险？
```

#### 输出示例

```text
校验结果：需要轻微修改

问题：
最终回答中出现“可靠人机协作”一词，但用户材料中没有人机协作相关经历，可能带来额外追问。

建议：
将“可靠人机协作”改为“大模型在科研辅助场景中的可靠性和可控性”，更贴合用户已有项目。
```

---

### 4.9 Training Planner：训练规划员

#### 核心任务

把一次复盘转化为下一次训练动作。

#### 输入

- 诊断结果；
- 材料召回率；
- 风险点；
- 最终回答；
- 用户弱点。

#### 输出

- 下次回答公式；
- 下一道建议训练题；
- 面试前准备清单；
- 可复用救场模板。

#### 不应该做

- 不重复诊断；
- 不重新生成长篇回答；
- 不泛泛鼓励。

#### 输出示例

```text
下一题建议：
请用 60 秒说明你在这个项目中具体负责了什么，以及哪些部分不是你完成的。

原因：
本次复盘显示你的主要风险是个人贡献边界不清。

下次救场模板：
这个项目中我主要负责的是 ____。
团队其他成员主要负责 ____。
我自己做得比较深入的部分是 ____。
如果继续改进，我认为可以从 ____ 入手。
```

## 5. 给 Agent 增加禁止项

很多时候，输出质量差不是因为模型不知道要做什么，而是因为它太愿意补全。因此，在保研面试场景中，禁止项比任务项更重要。

### 5.1 全局禁止项

所有 Agent 都应该遵守：

```text
不能编造用户没提供的经历；
不能默认用户读过某篇论文；
不能默认用户了解某个导师方向；
不能把课程作业包装成科研成果；
不能把团队成果说成个人贡献；
不能把“参与”改写成“主导”；
不能把工具使用包装成独立算法能力；
不能使用用户无法承接的高级术语；
不能在没有指标时写“显著提升”；
不能在没有证据时写“深入研究”。
```

### 5.2 Agent 特定禁止项

#### Material Analyst

- 不能评价回答；
- 不能擅自扩展材料；
- 不能把模糊经历提炼成确定成果。

#### Evidence Planner

- 不能根据用户回答反推应调用证据；
- 只能基于问题和材料判断“应该用什么”。

#### Evidence Mapper

- 不能将用户没有明确说出的内容算作已使用；
- 不能因为回答语气积极就判定证据充分。

#### Skeptical Professor

- 不能给泛泛鼓励；
- 不能输出“建议更具体”这种无指向建议；
- 必须指出具体风险表述和可能追问。

#### Synthesizer

- 不能追求华丽；
- 不能生成用户无法自然说出口的书面化回答；
- 不能为了提高亮点引入无证据方向。

#### Verifier

- 不能重写完整回答；
- 只负责检查、指出问题和建议修改。

## 6. 建立证据引用机制

### 6.1 为什么需要证据引用

证据引用机制是让产品从“AI 点评”升级为“材料驱动诊断”的关键。

如果系统说：

> 你应该提到项目经历。

用户会问：

> 为什么？

更好的方式是：

```text
结论：这道题应该提到项目经历。
依据：证据卡 #1 显示用户做过“大模型论文中译英 TeX 编辑器”，该项目能够解释用户为什么对大模型科研写作场景感兴趣。
```

### 6.2 内部结构建议

即使最终界面不展示复杂结构，内部也可以遵循：

```text
claim: 诊断结论
evidenceRefs: 引用的证据卡 ID
reason: 为什么这些证据支持该结论
missingInfo: 仍缺少的信息
confidence: high | medium | low
```

### 6.3 示例

```text
claim: 回答缺少项目证据
evidenceRefs: [card_1, card_2]
reason: 用户材料中已有论文中译英 TeX 编辑器项目和 Prompt 设计经历，但临场回答只表达了泛泛兴趣。
missingInfo: 无
confidence: high
```

### 6.4 用户可见表达

用户不需要看到复杂 JSON，只需要看到：

```text
你的回答缺少项目证据。

依据：
你在材料中提到过“大模型论文中译英 TeX 编辑器”和“Prompt 设计经历”，但临场回答没有使用这些内容，因此回答显得像模板表达。
```

## 7. 拆分 Evidence Planner 和 Evidence Mapper

### 7.1 拆分逻辑

Evidence Planner 解决：

> 这道题本来应该用哪些材料？

Evidence Mapper 解决：

> 用户实际用了哪些材料？

两者必须分开，因为这能避免模型被用户回答带偏。

### 7.2 工作流程

```text
问题 + 材料
↓
Question Intent Analyst 判断问题意图
↓
Evidence Planner 规划应调用证据
↓
用户回答
↓
Evidence Mapper 判断实际使用证据
↓
计算材料召回率
```

### 7.3 质量价值

这个拆分带来的最大好处是：

> “材料召回率”更可信。

因为系统不是看完用户回答后才随口说“你漏了什么”，而是先独立建立了应答标准。

### 7.4 示例

```text
问题：你为什么想继续做大模型与科研写作相关方向？

Evidence Planner：
应调用证据：
1. 论文中译英 TeX 编辑器项目；
2. Prompt 设计经历；
3. 科研写作中的术语一致性问题。

Evidence Mapper：
实际使用证据：
无。

材料召回率：
0 / 3

诊断：
用户不是没有内容，而是临场没有召回任何可支撑兴趣的材料。
```

## 8. 强化 Professor Agent：从风险标签到压力测试

### 8.1 当前常见问题

很多风险分析只输出：

```text
风险：证据不足
建议：补充具体细节
```

这不够有价值。

Professor 应该更像一个导师压力测试器。

### 8.2 标准输出结构

每个风险点都应包含：

```text
风险表述：
导师追问：
危险原因：
当前承接能力：
安全回应：
需要补充的信息：
```

### 8.3 示例一：贡献夸大风险

```text
风险表述：
我主要负责模型优化。

导师追问：
你优化的是模型参数、Prompt，还是推理流程？

危险原因：
“模型优化”容易被理解为算法或参数层面的优化。但用户材料中只提到 Prompt 设计和输出格式控制，如果不能解释清楚，会显得贡献夸大。

当前承接能力：
中等偏低。用户材料不足以支撑“模型优化”这个说法。

安全回应：
我这里说的优化主要是 Prompt 模板、输出格式约束和交互流程优化，不涉及模型参数训练。

需要补充的信息：
具体优化了哪些 Prompt 模板？如何判断优化有效？
```

### 8.4 示例二：研究方向过度包装风险

```text
风险表述：
我希望未来深入研究多模态智能体。

导师追问：
你读过哪些多模态智能体相关论文？你之前做过相关项目吗？

危险原因：
用户材料中只有大模型论文翻译项目，没有多模态智能体相关经历。如果直接说“深入研究”，容易被追问到无法承接。

当前承接能力：
低。

安全回应：
我目前的经历还比较偏大模型应用和科研写作场景，但也希望在研究生阶段逐步了解更复杂的人机协作和智能体系统。

需要补充的信息：
是否真正阅读过目标导师相关论文？是否有多模态或智能体相关基础？
```

## 9. 强化 Synthesizer：生成安全可承接回答

### 9.1 最终回答的目标

最终回答不是“最漂亮回答”，而是：

> **真实、具体、能承接追问、适合口述的安全回答。**

### 9.2 生成优先级

```text
真实性 > 可承接性 > 具体性 > 结构清晰 > 口述自然 > 高级感
```

### 9.3 生成过程建议

Synthesizer 在生成回答前，应先明确：

1. 这道题必须回答的核心点；
2. 可以使用的材料证据；
3. 不能使用或需要降级表达的风险点；
4. 用户能够承接的术语边界；
5. 回答长度要求。

### 9.4 输出格式

建议输出：

```text
30 秒版
60 秒版
使用证据
风险控制说明
如果被追问，可以这样补充
```

### 9.5 示例

```text
30 秒版：

我对这个方向的兴趣不是单纯因为大模型很热，而是来自我之前做过的一个论文中译英 TeX 编辑器项目。在这个项目中，我发现大模型在科研写作场景里虽然生成能力很强，但仍然容易出现术语不一致、格式被破坏和上下文不稳定的问题。因此我希望研究生阶段能进一步学习大模型在科研辅助场景中的可靠性和可控性问题。

使用证据：
- 论文中译英 TeX 编辑器项目；
- Prompt 设计经历；
- 科研写作中的术语一致性问题。

风险控制：
没有使用“深入研究多模态智能体”等用户材料无法支撑的表述。
```

## 10. 增加 Verifier / Critic 环节

### 10.1 为什么需要 Verifier

前面的诊断可能都很好，但最终回答仍可能出现问题：

- 生成了用户没提供的信息；
- 使用了过度高级的术语；
- 夸大了个人贡献；
- 没有正面回答题目；
- 听起来像书面稿，不适合口述。

Verifier 就是最终安全阀。

### 10.2 Verifier 的职责

Verifier 不负责生成新答案，只负责检查最终答案是否安全。

### 10.3 检查清单

```text
是否使用了未提供材料？
是否夸大个人贡献？
是否引入无法承接的术语？
是否正面回答题目？
是否保留具体证据？
是否适合口述？
是否存在新的追问风险？
是否需要降级某些表达？
```

### 10.4 输出示例

```text
校验结果：不完全通过

问题 1：
回答中提到“可靠人机协作”，但用户材料中没有相关经历，可能引出额外追问。

建议：
改为“大模型在科研辅助场景中的可靠性和可控性”。

问题 2：
回答中说“我深入研究了大模型应用”，但材料只说明用户做过应用项目。

建议：
改为“我初步接触到大模型应用中的一些实际问题”。
```

### 10.5 质量收益

加入 Verifier 后，最终回答会更稳，不容易出现“诊断严谨但答案冒进”的问题。

## 11. 增加置信度和信息缺口

### 11.1 为什么需要置信度

AI 在材料不足时也容易强行输出确定结论。  
保研场景中，这会带来误导。

因此每个关键 Agent 可以输出：

```text
confidence: high | medium | low
missingInfo: []
```

### 11.2 示例

```text
置信度：中

原因：
用户没有提供目标导师方向，因此关于“导师匹配度”的分析只能基于一般保研复试标准。

信息缺口：
1. 目标导师或实验室研究方向；
2. 用户项目中的具体个人贡献；
3. 项目是否有评价指标。
```

### 11.3 用户可见表达

```text
当前分析可信度：中

如果你补充目标导师方向和项目指标，系统可以更准确判断哪些表达适合使用。
```

### 11.4 产品价值

这会让产品显得更专业，也能减少模型胡说带来的风险。

## 12. 建立冲突解决规则

### 12.1 为什么需要冲突规则

不同 Agent 之间可能产生冲突：

- Evidence Planner 认为某材料很适合使用；
- Professor 认为该材料说出来有风险；
- Synthesizer 不知道该保留还是删除。

如果没有冲突解决规则，最终输出会不稳定。

### 12.2 推荐冲突规则

```text
如果“亮点”和“风险”冲突，优先保留真实且可承接的部分，删除或降级无法承接的高级表述。
```

### 12.3 具体规则表

| 冲突类型 | 处理原则 |
|---|---|
| 材料有亮点，但用户不熟 | 降级表达，不直接删除 |
| 回答更高级，但无证据 | 删除或标注需要补充 |
| 贡献表达有吸引力，但边界不清 | 改成边界更清晰的表达 |
| 导师匹配强，但用户不了解导师论文 | 改成“我初步关注到”，不要说“深入研究过” |
| 冷静回答更完整，但太长 | 压缩成 30 秒口述版 |
| 专业术语准确但用户无法解释 | 替换为更朴素的表达 |

### 12.4 示例

原表达：

```text
我希望未来深入研究多模态智能体与人机协作。
```

冲突：

- 亮点：方向听起来前沿；
- 风险：用户材料无法支撑。

处理后：

```text
我目前的经历还比较偏大模型应用和科研写作场景，但希望研究生阶段在可靠应用和人机协作相关问题上进一步学习。
```

## 13. 引入回答成熟度分级

### 13.1 为什么不用单纯分数

简单打分容易显得主观。  
成熟度分级更能体现训练路径。

### 13.2 推荐分级

```text
L1：模板表达
L2：有经历但缺少问题意识
L3：有经历 + 有问题意识
L4：有经历 + 问题意识 + 方向匹配
L5：有边界感 + 能承接追问 + 有未来计划
```

### 13.3 示例

用户临场回答：

```text
我对 AI 很感兴趣，觉得发展前景很好。
```

系统判断：

```text
当前成熟度：L1 模板表达

原因：
回答只有兴趣和前景判断，没有个人经历、具体问题或方向连接。
```

用户冷静回答：

```text
我之前做过一个大模型辅助论文翻译工具，在项目中发现大模型在科研写作场景里存在术语一致性和可控性问题，所以希望继续研究相关方向。
```

系统判断：

```text
当前成熟度：L3 有经历 + 有问题意识

原因：
回答补充了项目经历和具体问题，但还没有连接目标导师方向，也没有说明未来计划。
```

### 13.4 价值

成熟度分级能让用户看到自己从 L1 到 L3、L4 的进步，比一个简单分数更有教育意义。

## 14. 用标准样例持续调优

### 14.1 为什么需要样例

只靠感觉改 Prompt 很容易越改越长，却不一定更准。

更有效的方法是准备少量高质量样例，反复测试 Agent 输出。

### 14.2 建议准备 5 类样例

1. 动机类：为什么选择这个方向？
2. 项目类：介绍你的项目经历。
3. 贡献边界类：你具体负责什么？
4. 压力质疑类：这个项目有什么创新？
5. 未来规划类：研究生阶段想做什么？

### 14.3 每个样例应包含

```text
用户背景材料
面试问题
临场回答
冷静回答 / 多版本回答
理想证据卡
理想问题意图
理想风险点
理想安全回答
理想训练建议
```

### 14.4 样例评估标准

每次改 Prompt 或调整 Agent 后，用这些样例检查：

```text
是否识别出关键材料？
是否正确判断问题意图？
是否计算出合理材料召回？
是否指出真正风险？
是否避免编造和夸大？
最终回答是否可口述？
是否生成了可迁移模板？
```

### 14.5 价值

这比单纯增加 Prompt 字数更能提升质量。

## 15. Agent 输出不要全部展示给用户

### 15.1 问题

多 Agent 系统内部可能产生大量分析。如果全部展示给用户，报告会：

- 太长；
- 重复；
- 难以阅读；
- 抓不住重点；
- 降低产品体验。

### 15.2 建议采用分层展示

#### 第一层：用户第一眼必须看到

```text
一句话诊断
材料召回率
最大风险
最佳融合回答
下次救场模板
```

这一层应该放在报告顶部。

#### 第二层：用户可以展开查看

```text
问题真实意图
证据遗漏
导师追问风险
逐句诊断
多版本对比
回答成熟度
```

这一层适合做折叠面板或卡片。

#### 第三层：内部调试或高级用户查看

```text
每个 Agent 的完整输出
置信度
信息缺口
证据引用
Verifier 检查结果
冲突解决过程
```

这一层不应该默认展示。

### 15.3 用户报告推荐结构

```text
1. 一句话复盘结论
2. 材料召回率
3. 最大风险
4. 临场损失 / 多版本差异
5. 最佳融合回答
6. 救场模板 / 回答公式
7. 展开查看：详细诊断
```

### 15.4 示例

```text
一句话诊断：
你不是没有内容，而是临场没有调用自己的项目证据，导致回答像模板。

材料召回率：
0 / 3

最大风险：
回答缺少个人经历支撑，导师可能追问“你的兴趣具体来自哪里”。

最佳融合回答：
……
```

## 16. 质量优化优先级

### 16.1 第一优先级：Evidence Planner / Mapper

原因：

- 它决定产品是否真正“材料驱动”；
- 它支撑材料召回率；
- 它能让产品区别于普通 ChatGPT 点评。

核心优化：

```text
先判断应使用材料，再判断实际使用材料。
每个遗漏结论都绑定证据卡。
```

### 16.2 第二优先级：Skeptical Professor

原因：

- 它决定产品是否具备保研导师视角；
- 它能识别漂亮但危险的表达；
- 它能帮助用户提前准备追问。

核心优化：

```text
每个风险点输出：
风险表述 → 导师追问 → 危险原因 → 当前承接能力 → 安全回应。
```

### 16.3 第三优先级：Synthesizer + Verifier

原因：

- 用户最终最关心能不能拿走一个可用回答；
- 最终回答如果不安全，会严重影响产品可信度。

核心优化：

```text
Synthesizer 生成安全可承接回答；
Verifier 检查是否编造、夸大、答偏、不可承接。
```

### 16.4 第四优先级：Training Planner

原因：

- 它让产品从一次复盘变成训练闭环；
- 但相对容易实现，可以后续优化。

核心优化：

```text
根据最大风险推荐下一题和救场模板。
```

## 17. 推荐优化路线

### 17.1 完整路线

```text
第一步：定义所有 Agent 的输入、输出、禁止项；
第二步：建立证据卡机制，让关键结论绑定材料来源；
第三步：拆分 Evidence Planner 和 Evidence Mapper；
第四步：强化 Professor，让它输出追问、危险原因、安全回应；
第五步：重写 Synthesizer 目标，生成安全可承接回答，而不是漂亮答案；
第六步：增加 Verifier，只检查最终回答；
第七步：引入信息缺口和置信度；
第八步：用 3–5 个标准样例反复测试；
第九步：把最终报告做分层展示，避免信息过载。
```

### 17.2 如果只能做三件事

优先做：

```text
1. 所有关键结论绑定证据卡或信息缺口；
2. Professor 输出“追问 + 危险原因 + 安全回应”；
3. Synthesizer 生成后由 Verifier 检查真实性和可承接性。
```

### 17.3 如果只能做一件事

最优先做：

> **让每个关键诊断结论都绑定材料证据卡或指出信息缺口。**

这是产品从“AI 点评”升级为“基于材料的复盘诊断”的关键。

## 18. 可直接放入 Product Memo 的表述

可以在 Memo 中这样描述质量优化思路：

> 本项目没有把多 Agent 简单理解为“多个角色分别输出意见”，而是将保研面试复盘拆解为一套可验证的诊断流程。Material Analyst 先把用户材料整理成证据卡，Question Intent Analyst 判断题目真实考察目标，Evidence Planner 在看用户回答前规划本题应调用的材料，Evidence Mapper 再判断用户实际使用了哪些材料，由此得到材料召回率。Skeptical Professor 从导师视角审查回答中可能被追问的风险点，并给出安全回应。Answer Synthesizer 不追求最华丽的回答，而是在真实性、可承接性和口述自然度之间生成稳妥版本。最后，Verifier 检查最终回答是否编造、夸大或引入用户无法承接的术语。通过这种轻量多 Agent 诊断链，产品从简单 Prompt 点评升级为基于材料证据的结构化复盘系统。

---

## 19. 最终总结

Interview Replay 的功能完整性已经基本成立。下一阶段最重要的不是继续增加功能，而是提高判断可信度。

质量优化的核心可以概括为：

> **边界清晰、证据驱动、风险敏感、回答安全、输出分层。**

具体来说：

- Material 系列 Agent 负责“你有什么材料”；
- Evidence 系列 Agent 负责“这题该用什么、你实际用了什么”；
- Professor 负责“哪里会被导师刺穿”；
- Gap / Diff 负责“临场和冷静差在哪里”；
- Synthesizer 负责“怎么说得真实、具体、可承接”；
- Verifier 负责“有没有编造、夸大、答偏”；
- Planner 负责“下一次练什么”。

最终，产品不应该只是告诉用户：

> 你的回答不够具体。

而应该告诉用户：

> 你本来有这些材料可以支撑这道题，但临场没有召回；你的某个表述会被导师追问；更稳的说法应该保留真实经历、降低包装风险，并准备好下一句安全回应。

这才是 Interview Replay 相比普通 AI 面试工具和直接使用 ChatGPT 的核心质量优势。
