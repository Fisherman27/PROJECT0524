# Interview Replay 轻量前端优化方案

本文档根据当前代码完成情况重新调整前端优化方向。

当前项目的核心功能已经比较完整：材料预分析、问题预分析、双模式复盘、多 Agent 诊断、质量摘要、安全回答、折叠式报告、Markdown 导出和 Agent 侧边栏都已经具备。因此后续前端优化不应该继续做复杂大改版，而应该聚焦在“让用户更容易理解、操作更顺、Demo 更稳定”。

## 1. 当前前端完成情况

### 1.1 首页已经具备的能力

当前首页已经实现：

- 产品标题和简短定位；
- 面试背景信息填写；
- 背景材料输入；
- 材料文件上传和文件标签；
- 完整材料字数提示；
- “分析材料”按钮；
- 材料分析完成后的证据卡数量提示；
- 材料变更后的重新分析提示；
- 面试前模拟和面试后复盘入口；
- Agent Pipeline 侧边栏。

这说明首页已经不再是单纯入口页，而是具备了材料工作台雏形。

### 1.2 面试前页面已经具备的能力

当前面试前页面已经实现：

- 面包屑返回首页；
- 面试问题输入；
- AI 生成练习问题；
- 问题预分析状态；
- 作答时长设置；
- 5 秒准备倒计时；
- 限时临场作答；
- 提前锁定；
- 临场回答锁定；
- 冷静回答；
- 最终评审；
- 加载时 Agent Pipeline；
- 结果报告和真实 Agent trace。

这说明面试前页面已经有比较明确的训练流程。

### 1.3 面试后页面已经具备的能力

当前面试后页面已经实现：

- 面包屑返回首页；
- 真实问题输入；
- 问题预分析状态；
- 问题意图和建议调用材料展示；
- 多回答版本管理；
- 至少两个版本的提交限制；
- 最终评审；
- 加载时 Agent Pipeline；
- 结果报告和真实 Agent trace。

这说明面试后页面已经具备“真实复盘工作台”的主体能力。

### 1.4 报告页已经具备的能力

当前报告页已经实现：

- 顶部质量摘要；
- 最佳安全回答；
- 风险与压力测试；
- 材料召回率；
- 证据依据；
- 模式特有诊断；
- 训练建议；
- 复盘卡片；
- 问题真实意图；
- 回答安全校验；
- 材料证据库；
- 多角色诊断链；
- 折叠式报告区块；
- 复制和 Markdown 导出。

这说明报告页不需要重新设计信息架构，只需要增强“如何阅读报告”的提示和空状态。

## 2. 调整后的核心判断

原方案里提到的很多方向已经部分落地，继续大改会带来不必要的风险。

后续前端优化应该遵循：

```text
不重做页面结构
不引入复杂新依赖
不改后端 Agent 流程
不做重型动效或完整新设计系统
只补齐说明、引导、示例、空状态和 Demo 体验
```

最终目标是让用户和评委在不看说明文档的情况下，也能理解：

- 这个产品先分析我的材料；
- 为什么要先分析材料；
- 两个模式分别适合什么场景；
- 当前这一步我要填写什么；
- 系统正在调用哪些诊断角色；
- 报告应该先看哪里；
- 如果没有材料、问题或回答，下一步该怎么做。

## 3. 轻量优化范围

### 3.1 P0：必须做的轻量体验补强

P0 只做低风险、局部、对 Demo 提升明显的优化。

#### 3.1.1 首页增加“怎么使用”短说明

在首页标题下方或材料工作台前增加一个轻量说明区：

```text
使用流程
1. 先填材料：简历、项目、科研、课程、竞赛都可以。
2. 再分析材料：系统会提取可复用证据卡。
3. 最后选择模式：面试前练临场，面试后复盘真实回答。
```

注意：

- 不做长说明书；
- 不讲复杂技术架构；
- 文案要直接服务用户操作。

#### 3.1.2 首页增加材料准备提示

在背景材料输入区附近增加轻量 checklist。

建议检查项：

```text
是否包含项目/科研经历
是否写清个人贡献
是否提到目标方向
是否包含可量化结果或具体方法
```

实现方式：

- 用前端关键词和长度做简单判断；
- 不调用 AI；
- 不给分；
- 只提示“建议补充”或“已覆盖”。

#### 3.1.3 首页增加示例材料填充

为了 Demo 稳定，增加一个“载入示例材料”按钮。

点击后填充：

- 面试类型；
- 目标方向；
- 目标院校；
- 背景材料。

注意：

- 示例必须虚构和脱敏；
- 不新增数据库；
- 不新增 API；
- 示例数据可以放在前端常量文件中。

#### 3.1.4 模式卡片补充“适合你吗”

当前模式卡片已经有标题、描述和标签，但用户仍可能不知道该选哪个。

建议每个模式卡片增加两类短信息：

```text
适合：正式面试前训练 / 已经面试完复盘
你需要：一道问题 + 临场/冷静回答 或 多个回答版本
```

这只需要扩展 ModeCard 展示，不改变页面结构。

#### 3.1.5 pre/post 页面增加顶部步骤引导

在面试前和面试后页面标题下方增加简短步骤条。

面试前：

```text
1. 确认问题
2. 临场作答
3. 冷静重答
4. 生成复盘
```

面试后：

```text
1. 填写真实问题
2. 添加回答版本
3. 生成复盘
4. 提炼公式
```

实现方式：

- 静态步骤条即可；
- 不要求复杂动态激活；
- 可根据已有状态做简单 active 标记，但不是必须。

#### 3.1.6 表单局部增加填写提示

在关键输入区增加一两句具体提示。

例如：

```text
面试问题：尽量写导师原话，不要只写“项目介绍”。
临场回答：保留当时真实表达，不要事后润色。
冷静回答：补充刚才没来得及说的证据和逻辑。
回答版本：建议至少包含“真实回答”和“事后修改版”。
```

这些提示比抽象说明更能降低用户操作成本。

#### 3.1.7 Loading 文案增强

当前 LoadingState 已经有 spinner 和 Agent Pipeline。

建议轻量增强：

- LoadingState 支持传入阶段列表；
- pre/post 提交时展示“正在复用材料分析、匹配证据、模拟导师追问、生成安全回答”等短句；
- 不做真实流式进度；
- 不新增后端接口。

#### 3.1.8 报告页增加“阅读顺序提示”

在报告顶部质量摘要旁边或下方增加一段简短提示：

```text
建议先看“最佳安全回答”和“质量摘要”，再看材料召回、风险和详细诊断。
```

作用：

- 让评委知道报告不是随机堆叠；
- 让用户知道先看哪里。

#### 3.1.9 空状态和错误提示补齐

重点补齐：

- 首页没有材料时，引导填写或载入示例；
- 没有材料分析时进入 pre/post，提示“最终评审会自动补跑，但建议先回首页分析材料”；
- 问题分析失败时，说明最终复盘会自动补充；
- 报告某个列表为空时显示“本次未发现明显问题”或“材料不足，建议补充经历”。

## 4. P1：时间允许再做

P1 可以做，但不影响核心 Demo。

### 4.1 示例模式扩展

增加：

- 载入面试前示例；
- 载入面试后示例；
- 示例问题；
- 示例回答版本。

但这会触及多个页面状态，风险略高于首页示例材料，因此放 P1。

### 4.2 报告顶部 Demo 摘要

在报告顶部加入更像展示用的四格摘要：

```text
一句话诊断
材料召回
最高风险
下一步训练
```

当前已有 QualitySummaryCard，可只做轻微视觉强化，不重新设计报告。

### 4.3 轻量“为什么可信”说明

在报告末尾或 Agent Trace 上方增加：

```text
本报告由材料分析、问题意图、证据匹配、导师风险、回答融合和安全校验等角色共同生成。
```

这不是技术解释，而是帮助用户理解“为什么不是普通润色”。

### 4.4 移动端细节整理

当前页面以桌面 Demo 为主。若有时间，可以补：

- 侧边栏在窄屏下移动到正文下方；
- 模式卡片和报告区块间距调整；
- 长按钮文本换行保护。

## 5. 暂不建议做

以下内容暂不建议在当前阶段做：

- 新增完整 onboarding 弹窗流程；
- 新增复杂新手任务系统；
- 新增账号、历史记录或云端保存；
- 新增服务端流式 Agent 进度；
- 新增复杂动效；
- 重做首页视觉体系；
- 重做报告信息架构；
- 引入大型 UI 组件库；
- 引入图表库只为了展示轻量指标；
- 把侧边栏改成复杂可交互调试台。

理由：

- 当前产品功能完整性已经够强；
- 比赛 Demo 更需要清晰和稳定；
- 复杂交互会增加状态同步风险；
- 当前代码已有较多未提交改动，继续大改不划算。

## 6. 推荐实现优先级

建议实现顺序：

```text
1. 首页使用说明 + 材料准备提示
2. 载入示例材料
3. 模式卡片补充“适合/需要”
4. pre/post 顶部步骤引导
5. 表单局部填写提示
6. Loading 文案增强
7. 报告阅读顺序提示
8. 空状态和错误提示补齐
```

如果时间非常紧，只做前四项即可。

## 7. 验收标准

完成轻量前端优化后，应满足：

- 用户进入首页 5 秒内知道产品怎么用；
- 用户知道为什么要先分析材料；
- 用户知道两个模式怎么选；
- 用户不准备材料也能通过示例材料快速 Demo；
- pre/post 页面每一步都有明确动作提示；
- 等待最终评审时，用户知道系统在做哪些诊断；
- 报告顶部告诉用户先看哪里；
- 空状态和失败状态不让用户卡住；
- 不新增复杂依赖；
- 不改变后端 Agent 流程；
- 不暴露 API Key；
- 不引入数据库或账号系统。

## 8. 最终判断

当前不需要再做复杂前端升级。最值得做的是“轻量说明 + 操作引导 + 示例填充 + 报告阅读提示”。

这类优化实现成本低、工程风险小，但能显著改善 Demo 观感，让产品从“功能很多的 AI 表单”变成“用户能顺着完成一次训练的面试复盘工作台”。
